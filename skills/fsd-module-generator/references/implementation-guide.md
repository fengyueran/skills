# 实现规范指南

## 目录
- 1. 目标
- 2. UI 组件化
- 3. 单一职责
- 4. 可测试性
- 5. 推荐拆分阈值
- 6. PR 自检清单

## 1. 目标

本指南约束“这一层里怎么写得好”，不替代 FSD 分层规则。

- FSD 解决“代码放哪层”
- 本指南解决“这层里如何实现”

## 2. UI 组件化

- 页面/组件文件只承担编排与展示，不堆积复杂业务分支。
- 把可复用 UI 抽成独立组件，优先放当前 slice 的 `ui/`。
- 组件优先通过 props 驱动，避免隐式读取过多外部状态。
- 同目录中组件数量增多时，拆分为：
  - `ui/index.ts`（公共导出）
  - `ui/<slice-or-scene-name>.tsx`（主视图，语义化命名）
  - `ui/components/*.tsx`（子组件，或按场景命名为 `tabs/`、`panels/`、`fields/`）

## 3. 单一职责

- 一个函数/组件只处理一个核心意图。
- 视图与副作用分离：
  - 视图：`ui/`
  - 状态与业务流程：`model/`
  - IO 与请求：`api/`
- 避免“万能组件/万能 hook”：当出现多种独立场景分支时拆分。

## 4. 可测试性

- 纯逻辑尽量放 `model/` 或 `lib/`，避免耦合 DOM。
- 副作用通过注入或边界封装，便于 mock。
- 选择器保持纯函数，输入输出稳定。
- 组件测试聚焦行为：
  - 给定状态 -> 渲染结果
  - 用户交互 -> 调用 action/callback
  - 异步状态 -> loading/success/error 切换

## 5. 推荐拆分阈值

满足任一项就应考虑拆分：

- 单文件超过 200 行
- JSX 嵌套超过 4 层
- `if/switch` 分支超过 5 处
- 一个组件同时处理布局、业务流程、网络请求
- 测试中需要 mock 太多依赖才能跑通

## 6. PR 自检清单

- [ ] UI 是否已组件化，避免单文件过重
- [ ] 业务逻辑是否下沉到 `model`，请求是否在 `api`
- [ ] 是否存在“单一职责”违背点（万能组件/万能 hook）
- [ ] 关键分支是否可单测（至少覆盖 loading/error/success）
- [ ] 对外 API 是否清晰，导出是否最小化
